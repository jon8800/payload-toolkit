  <action>
The `reference` prop shape changes from `string` to a polymorphic relationship object. When populated by Payload, it looks like:
```typescript
{ relationTo: 'pages' | 'posts', value: { slug: string } | string }
```
When not populated, `value` is a string (the ID). When populated (depth >= 1), `value` is an object with `slug`.

1. Create a shared helper function `resolveHref` (can be inline in each component or a small utility) that:
   - Takes `type`, `url`, and `reference` (the relationship object)
   - For external: returns `url`
   - For internal: extracts the slug from the reference
     - If `reference.value` is an object with `slug`, use it
     - If `reference.relationTo` is 'posts', prefix with `/blog/`
     - If `reference.relationTo` is 'pages', prefix with `/`
     - Fallback to `#` if reference is missing or value is a string (not populated)

2. Update `src/blocks/Link/component.tsx`:
   - Change `reference` prop type from `string` to `{ relationTo?: string; value?: { slug?: string } | string }` (or a named type)
   - Replace `const href = type === 'external' ? url : reference || '#'` with the resolveHref logic
   - Use Next.js `Link` component from 'next/link' instead of plain `<a>` tag for internal links (better client-side navigation). Keep `<a>` for external links.

3. Update `src/blocks/Button/component.tsx`:
   - Same reference prop type change
   - Same href resolution logic
   - Same Next.js Link usage for internal, `<a>` for external

Keep all existing styling, className, and style logic unchanged. Only the href resolution and link rendering changes.
