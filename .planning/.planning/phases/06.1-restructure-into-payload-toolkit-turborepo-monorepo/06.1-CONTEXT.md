# Phase 6.1: Restructure into payload-toolkit Turborepo Monorepo - Context

**Gathered:** 2026-03-15
**Status:** Ready for planning

<domain>
## Phase Boundary

Restructure the two separate repos (payload-starter + create-payload-starter) into a single Turborepo monorepo called `payload-toolkit`. Create new GitHub repo, set up pnpm workspaces + Turborepo, move all code into the correct package structure, update the CLI scaffold logic, and archive the old repos.

</domain>

<decisions>
## Implementation Decisions

### Package Structure
- Turborepo standard layout: `apps/` for deployable things, `packages/` for libraries/tools
- `apps/starter/` — the Payload CMS starter template (currently the payload-starter repo root)
- `packages/create-payload-starter/` — the CLI scaffold package (currently the create-payload-starter repo)
- `packages/shared/` — shared utilities used by both (create-database, generate-env, etc.)
- Root `turbo.json` for pipeline configuration
- Root `pnpm-workspace.yaml` for workspace definition

### CLI Template Source
- Claude's discretion on the most practical approach (sparse clone + extract, release tarball, or shallow clone)
- Must work for end users running `npx @jon8800/create-payload-starter` — they get the template from GitHub

### GitHub Repo Strategy
- Create new repo `jon8800/payload-toolkit` (public)
- Archive old repos `jon8800/payload-starter` and `jon8800/create-payload-starter`
- Fresh git history in the monorepo (clean start)

### Claude's Discretion
- Turbo pipeline configuration (which tasks to cache, dependencies between packages)
- How the CLI extracts the template from the monorepo (sparse checkout, degit subdirectory, release tarball)
- Shared package contents and API
- Whether apps/starter needs its own turbo tasks or just inherits
- tsconfig project references setup

</decisions>

<specifics>
## Specific Ideas

- User consistently uses Turborepo for monorepo setups
- The monorepo pattern follows Payload's official repo structure (packages alongside create-payload-app)
- The CLI package will be published to npm/GitHub Packages — needs proper build + publish pipeline in turbo
- apps/starter/ should still be independently buildable and runnable (it's a real Next.js app)

</specifics>
