# Phase 6.1: Restructure into payload-toolkit Turborepo Monorepo - Research

**Researched:** 2026-03-15
**Domain:** Turborepo monorepo setup, pnpm workspaces, CLI template extraction
**Confidence:** HIGH

## Summary

This phase restructures two separate repos (payload-starter + create-payload-starter) into a single Turborepo monorepo called `payload-toolkit`. The Turborepo + pnpm workspaces approach is well-documented, mature, and widely used. The critical technical decision is how the CLI extracts the template subdirectory from the monorepo -- Payload's own `create-payload-app` uses GitHub's codeload tarball with the `tar` package to filter and extract a subdirectory, which is the proven pattern to follow.

Turborepo recommends an "Internal Packages" pattern for shared TypeScript packages (no build step, point directly to source), but since the CLI is published to npm and must work standalone, the shared package needs a proper build step. The starter app (apps/starter/) is a standard Next.js app that needs no Turborepo-specific changes beyond being in the workspace.

**Primary recommendation:** Use GitHub codeload tarball extraction (like Payload's own CLI) for template downloads, Turborepo Internal Packages pattern for shared utils consumed only within the monorepo, and a standard tsc build for the CLI package that gets published to npm.

### Pattern 1: GitHub Codeload Tarball Extraction (for CLI template download)
**What:** Download the monorepo tarball from GitHub and extract only the `apps/starter/` subdirectory
**When to use:** When the CLI needs to scaffold a new project from the template
**Why:** This is exactly how Payload's official `create-payload-app` works. Zero extra dependencies beyond `tar` (which is a small, well-maintained package). Works without git installed on the user's machine.

**Example:**
```typescript
// Source: Payload CMS create-payload-app pattern
import { Readable } from 'node:stream'
import { x as extractTar } from 'tar'

const repoUrl = 'https://codeload.github.com/jon8800/payload-toolkit/tar.gz/main'
const templatePath = 'payload-toolkit-main/apps/starter/'

const response = await fetch(repoUrl)
if (!response.ok || !response.body) {
  throw new Error(`Failed to download template: ${response.statusText}`)
}

await Readable.fromWeb(response.body as any).pipe(
  extractTar({
    cwd: targetDir,
    filter: (path) => path.startsWith(templatePath),
    strip: 2, // Remove "payload-toolkit-main/apps/starter/" prefix
  })
)
```

**Key details:**
- GitHub codeload URL format: `https://codeload.github.com/{owner}/{repo}/tar.gz/{branch}`
- The tarball root directory is `{repo}-{branch}/` (e.g., `payload-toolkit-main/`)
- `strip: 2` removes the two directory levels (`payload-toolkit-main/apps/`) but since we want `starter/` contents directly, strip should match the depth to the template folder
- Actually strip count = number of path segments to remove. For `payload-toolkit-main/apps/starter/src/...` to become `src/...`, strip = 3

### Pattern 3: turbo.json Pipeline Configuration
**What:** Define task dependencies and caching
**When to use:** Root of the monorepo

**Example:**
```json
{
  "$schema": "https://turbo.build/schema.json",
  "tasks": {
    "build": {
      "dependsOn": ["^build"],
      "outputs": ["dist/**", ".next/**"]
    },
    "dev": {
      "persistent": true,
      "cache": false
    },
    "lint": {},
    "typecheck": {
      "dependsOn": ["^build"]
    }
  }
}
```

**Notes:**
- `build` uses `"dependsOn": ["^build"]` so shared package builds before CLI
- `dev` is persistent (long-running) and not cached
- `lint` has no dependencies (can run in parallel)
- The starter app (apps/starter) has `build` producing `.next/**`; the CLI has `build` producing `dist/**`

### Pattern 4: Shared Package as Internal Package (no build step)
**What:** Shared package exports raw TypeScript, consumers transpile it
**When to use:** When the shared package is ONLY consumed within the monorepo

**However:** Since the CLI is published to npm and runs standalone, the shared package must be built before the CLI is built. Use the standard build approach:

```json
// packages/shared/package.json
{
  "name": "@jon8800/shared",
  "version": "0.1.0",
  "type": "module",
  "main": "./dist/index.js",
  "types": "./dist/index.d.ts",
  "exports": {
    ".": {
      "types": "./dist/index.d.ts",
      "default": "./dist/index.js"
    }
  },
  "scripts": {
    "build": "tsc",
    "typecheck": "tsc --noEmit"
  }
}
```

**Alternative (Internal Package, if shared is only used in dev):** If the starter app also wants to use shared utils at dev time, it can use the Internal Packages pattern (`"main": "./src/index.ts"`) with Next.js `transpilePackages`. But the CLI needs built output. A hybrid approach is possible: export both raw source and built output.

**Recommendation:** Keep it simple. Build the shared package with tsc. Both the CLI (at build time) and starter (at dev time via transpilePackages) consume the built output.

### Anti-Patterns to Avoid
- **Nested workspace patterns:** Turborepo does not support `apps/**` -- use `apps/*` only
- **TypeScript project references:** Turborepo explicitly advises against them; they add configuration overhead without benefit when Turbo handles the build graph
- **Root tsconfig extending into packages:** Each package should have its own tsconfig.json
- **Importing app code from shared packages:** Dependencies should flow from shared -> app, never reverse
- **Publishing internal packages:** If using Internal Packages pattern, those packages cannot be published to npm

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Template download from monorepo | Custom git sparse checkout logic | GitHub codeload tarball + tar package | Payload CMS uses this exact pattern; handles auth, errors, and extraction cleanly |
| Monorepo task orchestration | Custom build scripts | turbo run | Handles dependency graph, caching, parallelization automatically |
| Workspace linking | Manual symlinks | pnpm workspace: protocol | Automatic with pnpm-workspace.yaml |
| CLI bundling (if needed) | Custom rollup config | tsup | Single-command TypeScript bundling for npm packages |

**Key insight:** The heaviest lift in this phase is restructuring files and updating imports -- the tooling (Turborepo, pnpm workspaces) handles everything else automatically once configured.

## Common Pitfalls

### Pitfall 1: Tarball Path Prefix Mismatch
**What goes wrong:** The GitHub codeload tarball wraps files in `{repo}-{branch}/` prefix. If you miscalculate the strip count, files extract to wrong locations.
**Why it happens:** The prefix changes based on repo name and branch.
**How to avoid:** Always construct the filter path dynamically: `${repoName}-${branch}/apps/starter/`. Test with a known file.
**Warning signs:** Empty project directory after extraction, or nested directories appearing.

### Pitfall 2: Package Name Conflicts
**What goes wrong:** The starter template's package.json says `"name": "payload-starter"` but in the monorepo workspace, pnpm expects unique names.
**Why it happens:** Both apps/starter and the old repo share the same package name.
**How to avoid:** Keep `"name": "payload-starter"` for apps/starter (it is unique within the workspace). Do NOT scope it with `@jon8800/` since end users will get it as their project scaffold.
**Warning signs:** pnpm workspace install errors about duplicate names.

### Pitfall 3: CLI Templates Directory Path
**What goes wrong:** The CLI resolves templates relative to `__dirname`. After moving into the monorepo, the relative path may change.
**Why it happens:** `render-templates.ts` uses `path.resolve(__dirname, '..', 'templates')` which works when CLI is at repo root but may break in monorepo.
**How to avoid:** Keep the same relative structure (templates/ at CLI package root). The `__dirname` approach still works since it resolves from the built dist/ output.
**Warning signs:** "template not found" errors at runtime.

### Pitfall 4: Shared Package Import Paths
**What goes wrong:** Changing from relative imports (`./lib/create-database.js`) to package imports (`@jon8800/shared`) requires updating all import statements.
**Why it happens:** Code that used to be co-located is now in a separate package.
**How to avoid:** Use workspace protocol in package.json: `"@jon8800/shared": "workspace:*"`, update all imports in the CLI package.
**Warning signs:** Build errors about missing modules.

### Pitfall 5: Forgetting to Exclude Planning Files from Template
**What goes wrong:** The `.planning/` directory and other dev-only files end up in scaffolded projects.
**Why it happens:** The tarball extraction downloads everything under `apps/starter/`.
**How to avoid:** Keep `.planning/` at the monorepo root only (it is already listed in the scaffold cleanup). Add a `.templateignore` or handle in the filter function. Actually, since `.planning/` is at the monorepo root and the template is `apps/starter/`, this is already handled -- `.planning/` won't be in the starter subdirectory.
**Warning signs:** Users getting planning files in their projects.

### Pitfall 6: pnpm onlyBuiltDependencies Must Be at Root
**What goes wrong:** The starter's `pnpm.onlyBuiltDependencies` config for sharp/esbuild may not work in the workspace.
**Why it happens:** pnpm workspace configuration can behave differently than standalone.
**How to avoid:** Move `pnpm.onlyBuiltDependencies` to the root package.json for workspace-wide effect. Also keep it in apps/starter for when it is extracted as a standalone project.

## Code Examples

### Complete scaffold.ts Rewrite (tarball approach)
```typescript
// Source: Payload CMS create-payload-app pattern adapted for this project
import * as p from '@clack/prompts'
import fs from 'node:fs'
import path from 'node:path'
import { Readable } from 'node:stream'
import { x as extractTar } from 'tar'

const REPO_OWNER = 'jon8800'
const REPO_NAME = 'payload-toolkit'
const BRANCH = 'main'
const TEMPLATE_PATH = 'apps/starter'

export async function scaffoldProject(targetDir: string): Promise<void> {
  const s = p.spinner()

  if (fs.existsSync(targetDir)) {
    const files = fs.readdirSync(targetDir)
    if (files.length > 0) {
      p.log.error(`Directory "${targetDir}" already exists and is not empty.`)
      process.exit(1)
    }
  }

  // Ensure target directory exists
  fs.mkdirSync(targetDir, { recursive: true })

  s.start('Downloading starter template...')

  const tarballUrl = `https://codeload.github.com/${REPO_OWNER}/${REPO_NAME}/tar.gz/${BRANCH}`
  const prefix = `${REPO_NAME}-${BRANCH}/${TEMPLATE_PATH}/`

  try {
    const response = await fetch(tarballUrl)
    if (!response.ok || !response.body) {
      throw new Error(`HTTP ${response.status}: ${response.statusText}`)
    }

    const readable = Readable.fromWeb(response.body as any)

    await new Promise<void>((resolve, reject) => {
      readable
        .pipe(
          extractTar({
            cwd: targetDir,
            filter: (filePath: string) => filePath.startsWith(prefix),
            strip: prefix.split('/').filter(Boolean).length,
          })
        )
        .on('finish', resolve)
        .on('error', reject)
    })

    s.stop('Template downloaded.')
  } catch (err) {
    s.stop('Download failed.')
    p.log.error(`Failed to download template: ${err instanceof Error ? err.message : String(err)}`)
    process.exit(1)
  }

  // Remove setup-only files
  const filesToDelete = ['scripts/setup.ts', 'scripts/lib/modify-config.ts']
  for (const file of filesToDelete) {
    const filePath = path.join(targetDir, file)
    if (fs.existsSync(filePath)) {
      fs.rmSync(filePath, { recursive: true, force: true })
    }
  }

  p.log.success('Project scaffolded successfully.')
}
```

## Open Questions

1. **Should `pg` dependency stay in the CLI or move entirely to shared?**
   - What we know: Both create-database.ts and generate-env.ts are moving to shared. pg is only needed by create-database.
   - What's unclear: Whether the starter app also uses pg directly for seed scripts.
   - Recommendation: Move pg to shared's dependencies. The CLI gets it transitively via workspace linking. When published, pnpm will bundle the dependency correctly.

2. **Should the CLI package name change from `@anthropic/create-payload-starter` to `@jon8800/create-payload-starter`?**
   - What we know: Current package.json says `@anthropic/create-payload-starter` (likely placeholder). User's GitHub is jon8800.
   - What's unclear: Whether user wants scoped or unscoped name for npx usage.
   - Recommendation: Use `@jon8800/create-payload-starter` for consistency with the GitHub org. npx usage would be `npx @jon8800/create-payload-starter`.

3. **What pnpm version to pin in packageManager field?**
   - What we know: Starter's engines say `"pnpm": "^9 || ^10"`. Both are compatible.
   - Recommendation: Use `"packageManager": "pnpm@10.0.0"` (or whatever version is installed locally) and keep engines flexible.
